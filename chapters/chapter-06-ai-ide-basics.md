## 故事：小张的Agent模式体验

### 周一：告别"Tab依赖"

小张用Cursor已经半年了。从最初的惊艳——"按Tab就能生成代码"——到现在的习以为常，他对AI编程有了新的理解。

"Tab补全确实快，"他心想，"但它有个致命问题：我得一行行看着它生成，一行行按Tab。写一个新功能，手指头比脑子还累。"

周一早上，产品经理又扔来一个需求："给后台管理系统加个权限管理模块，要能动态分配角色和权限。"

按以前的做法，小张需要：
1. 先写数据模型（用户、角色、权限）
2. 写API接口（CRUD）
3. 写前端页面（列表、表单）
4. 写权限检查逻辑
5. 联调测试

"至少得一整天吧，"他叹了口气，"而且权限逻辑最容易出bug。"

这时，他注意到Cursor界面右上角有个新按钮：**Agent Mode**。

"这是什么？试试。"

### 1:00 PM 第一次用Agent模式

小张切换到Agent模式，界面变成了对话形式。

他输入：

```
给项目添加一个完整的RBAC权限管理系统：
1. 后端：用户、角色、权限三个模型，支持多对多关系
2. 前端：角色管理页面、权限分配页面
3. 实现权限检查中间件
4. 使用项目中已有的技术栈（React + Node.js + MongoDB）
```

Cursor没有立刻生成代码，而是先**制定计划**：

```
我将为你实现完整的RBAC权限管理系统。以下是我的执行计划：

**阶段1：后端实现**
- [ ] 创建 models/Role.js - 角色模型
- [ ] 创建 models/Permission.js - 权限模型  
- [ ] 创建 models/User.js - 更新用户模型，添加角色关联
- [ ] 创建 middleware/rbac.js - 权限检查中间件
- [ ] 创建 routes/roles.js - 角色管理API
- [ ] 创建 routes/permissions.js - 权限管理API

**阶段2：前端实现**
- [ ] 创建 pages/Roles/index.jsx - 角色列表页面
- [ ] 创建 pages/Roles/Edit.jsx - 角色编辑/创建页面
- [ ] 创建 pages/Permissions/index.jsx - 权限列表页面
- [ ] 更新 router.jsx - 添加新路由

**阶段3：验证**
- [ ] 检查所有文件是否正确创建
- [ ] 验证模型关系是否正确

预计修改文件数：10+
是否开始执行？(确认/修改计划/取消)
```

小张瞪大了眼睛。这不是直接生成代码，而是**先规划，再执行**。

他点击"确认"。

Cursor开始自动工作。他看着屏幕，一行行代码被写入文件，文件树不断刷新。几分钟后，所有文件都创建好了。

"这就...完了？"小张不敢相信。

他打开生成的代码检查：

**Role模型**：
```javascript
const roleSchema = new mongoose.Schema({
  name: { type: String, required: true, unique: true },
  description: { type: String },
  permissions: [{ type: mongoose.Schema.Types.ObjectId, ref: 'Permission' }],
  createdAt: { type: Date, default: Date.now }
});
```

**权限检查中间件**：
```javascript
const checkPermission = (requiredPermission) => {
  return async (req, res, next) => {
    try {
      const user = await User.findById(req.userId).populate({
        path: 'roles',
        populate: { path: 'permissions' }
      });
      
      const hasPermission = user.roles.some(role =>
        role.permissions.some(perm => perm.code === requiredPermission)
      );
      
      if (!hasPermission) {
        return res.status(403).json({ error: '没有权限' });
      }
      
      next();
    } catch (error) {
      res.status(500).json({ error: '权限检查失败' });
    }
  };
};
```

**前端角色管理页面**：
完整实现了列表展示、新增、编辑、删除功能，使用了项目中已有的组件库。

小张看完，深吸一口气。

"这比我手写的还好...而且只用了5分钟。"

### 周二：深入理解Agent模式

周一开始用Agent模式后，小张决定深入了解这个新功能。

他发现Agent模式有三个**核心特点**：

#### 1. 自主规划

传统Tab模式：你写注释，AI补全代码，你得一行行看着。

Agent模式：你描述需求，AI先分析、制定计划，你确认后才执行。

```
用户：优化这个函数的性能

AI分析：
"我分析了processData函数，发现以下问题：
1. 使用了O(n²)的嵌套循环
2. 每次循环都重新计算相同值
3. 可以考虑使用Map优化查找

优化方案：
- 使用Map存储中间结果，将查找降为O(1)
- 提前计算不变量
- 预计性能提升80%

是否执行优化？"
```

#### 2. 多文件联动

传统模式只能处理单个文件。Agent模式可以同时修改多个文件，并保持它们之间的一致性。

比如添加一个新功能：
- 后端：修改模型、添加路由、更新中间件
- 前端：创建页面、更新路由、添加API调用

Agent模式会自动处理这些文件的依赖关系。

#### 3. 自主验证

执行完成后，Agent会主动验证结果：

```
执行完成。正在验证结果...

✅ 所有文件创建成功
✅ 语法检查通过
✅ 依赖关系正确
⚠️ 发现1个潜在问题：
   - routes/roles.js 第45行缺少错误处理
   
是否修复这个问题？
```

### 周三：Plan模式与YOLO模式

小张发现Cursor的Agent模式还有两种子模式：

#### Plan模式

适合复杂任务，可控性更强。

```
用户：重构整个项目的错误处理

AI制定详细计划：
1. 创建统一的错误类 (src/errors/)
2. 修改所有API路由的错误处理
3. 更新前端错误捕获逻辑
4. 添加错误日志记录
5. 编写测试用例

每个步骤都会征求你的确认，你可以：
- 批准执行
- 修改计划
- 跳过某一步
- 取消整个任务
```

#### YOLO模式

适合简单、可回滚的任务，速度快。

```
用户：给所有console.log添加时间戳前缀

AI：直接执行，不需要每一步确认

（30秒后）
完成。修改了15个文件，添加了时间戳前缀。
```

小张总结：
- **Plan模式**：复杂任务，需要把控，安全第一
- **YOLO模式**：简单任务，追求速度，可回滚即可

### 周四：MCP协议扩展

周四下午，小张在文档里看到了一个新概念：**MCP**（Model Context Protocol）。

"这是什么？"

他研究了一下，发现MCP是Cursor支持的一种扩展协议，允许AI调用外部工具：

```
通过MCP，Cursor Agent可以：
- 查询数据库
- 调用外部API
- 读取本地文件系统
- 执行Shell命令
```

他配置了一个简单的MCP服务器，连接到了公司的测试数据库。

然后他问Agent：

```
查看users表里有多少用户，然后生成一个用户统计页面
```

Agent执行了：
1. 通过MCP查询数据库：`SELECT COUNT(*) FROM users`
2. 获取结果：1523个用户
3. 生成用户统计页面，包含真实数据

"这太强大了，"小张惊叹，"AI不再只是'看'代码，它可以'操作'真实世界。"

### 周五：正确使用Agent模式

周五，小张总结了一周的使用经验，写下了**Agent模式的最佳实践**：

#### ✅ 正确使用方式

**1. 明确需求，提供上下文**

```
❌ 不好的描述：
"优化这个页面"

✅ 好的描述：
"优化UserList页面的性能。当前页面加载时有明显的卡顿，
列表有1000+条数据。使用虚拟列表或分页优化，
保持现有的筛选和排序功能。"
```

**2. 审查计划，不要盲信**

```
AI生成的计划：
- [ ] 修改数据库模型
- [ ] 更新API接口
- [ ] 修改前端页面

你的审查：
"等等，修改数据库模型会影响线上数据吗？
先让我看看具体的修改方案。"
```

**3. 分阶段确认，保持控制**

复杂任务拆分成多个小任务，每个小任务执行完都检查：

```
Task 1: 创建数据库模型 ✓ 确认无误
Task 2: 编写API接口 ✓ 确认无误
Task 3: 编写前端页面 ✓ 确认无误
Task 4: 联调测试 ✓ 确认无误
```

**4. 验证结果，特别是安全性**

```
AI："任务完成。添加了用户认证功能。"

你：检查生成的代码
- ✅ JWT验证逻辑正确
- ✅ 密码加密使用了bcrypt
- ⚠️ token过期时间设置太长（7天）
  → 要求修改："token过期时间改为2小时"
```

#### ❌ 错误使用方式

**1. 完全放手，不审查**

```
用户："重构整个项目"
AI：执行中...
用户：喝咖啡去了，完全不看
（回来后发现项目崩了）
```

**2. 描述模糊，期望AI猜**

```
用户："改进一下"
AI："请问您想改进什么方面？"
用户："就是改进啊"
（AI只能瞎改一通）
```

**3. 忽视业务规则**

```
AI生成的权限系统逻辑正确，但：
- 没有考虑公司的特殊权限规则
- 没有对接现有的用户系统
（上线后业务逻辑混乱）
```

## 理论知识：Agent模式深度解析

### 什么是Agent模式？

Agent模式是AI编程工具的**第三次进化**：

| 阶段 | 代表工具 | 交互方式 | 用户角色 |
|:---|:---|:---|:---|
| **补全时代** | Copilot | Tab补全 | 主导者，AI辅助 |
| **对话时代** | Cursor Chat | 对话生成 | 主导者，AI执行单任务 |
| **Agent时代** | Cursor Agent, Claude Code | 自主规划-执行-验证 | 指挥官，AI自主执行 |

### Agent模式的核心能力

```
┌─────────────────────────────────────────┐
│            Agent模式工作流               │
├─────────────────────────────────────────┤
│                                         │
│  1. 需求理解                             │
│     ↓ 解析用户描述，识别任务目标          │
│                                         │
│  2. 上下文分析                            │
│     ↓ 读取项目结构、相关文件、依赖关系    │
│                                         │
│  3. 计划制定                              │
│     ↓ 分解任务，列出步骤，预估影响范围    │
│                                         │
│  4. 人工确认                              │
│     ↓ 用户审查计划，确认或修改            │
│                                         │
│  5. 自主执行                              │
│     ↓ 按计划修改文件、运行命令            │
│                                         │
│  6. 结果验证                              │
│     ↓ 检查语法、运行测试、验证结果        │
│                                         │
│  7. 人工审查                              │
│     ↓ 用户审查修改，确认或回滚            │
│                                         │
└─────────────────────────────────────────┘
```

### Cursor Agent vs Claude Code

| 维度 | Cursor Agent | Claude Code |
|:---|:---|:---|
| **界面** | IDE集成（图形化） | 纯命令行 |
| **上下文** | 通过@引用文件 | 自动分析整个代码库 |
| **交互** | 点击确认/修改 | 输入y/n/revise |
| **MCP支持** | 需要配置 | 原生支持 |
| **适用场景** | 日常开发、复杂任务 | 大型重构、自动化脚本 |
| **学习曲线** | 较低 | 较高 |

### 什么时候用Agent模式？

| 场景 | 推荐模式 | 原因 |
|:---|:---|:---|
| 写单个函数 | Tab补全 | 快速，不需要规划 |
| 解释代码/问问题 | Chat模式 | 对话交互更合适 |
| 修改多个文件 | Agent模式 | 自主处理依赖关系 |
| 大型重构 | Agent模式/Claude Code | 需要规划和批量执行 |
| 自动化任务 | Agent模式 + MCP | 可以调用外部工具 |

## 实践案例：Agent模式工作流

### 案例1：添加新功能（多文件修改）

**任务**：添加"用户评论"功能

**传统方式**：
1. 手动创建Comment模型
2. 手动编写API路由
3. 手动创建前端组件
4. 手动更新相关页面
5. 手动测试联调

**Agent模式**：

```
用户：添加用户评论功能，包含：
- 评论模型（关联用户和内容）
- 评论的增删改查API
- 前端评论列表和表单
- 使用项目中已有的组件和API规范

AI：制定计划...
      阶段1：后端 (4个文件)
      阶段2：前端 (3个文件)
      阶段3：验证
      
用户：确认计划

AI：执行阶段1...完成
用户：检查后端代码 ✓

AI：执行阶段2...完成
用户：检查前端代码 ✓

AI：执行阶段3...完成
用户：运行测试 ✓

总耗时：15分钟（vs 传统方式2小时+）
```

### 案例2：代码重构

**任务**：将回调式代码改为async/await

**Agent模式执行**：

```
用户：将src/services/下的所有回调式API调用改为async/await

AI：
1. 分析src/services/目录，发现5个文件需要修改
2. 制定计划：
   - 修改userService.js
   - 修改orderService.js
   - ...
3. 用户确认
4. 逐文件修改，保持API接口不变
5. 验证语法正确
6. 询问是否运行测试

用户：运行测试
AI：执行npm test...
      所有测试通过 ✓
```

### 案例3：Bug修复（多步骤）

**任务**：修复"用户偶尔登录失效"的问题

```
用户：用户反馈偶尔会自动登出，帮我排查原因

AI：
1. 分析认证相关代码...

   - token过期时间不一致（前端2小时，后端1小时）
   - 刷新token逻辑有race condition

   - [ ] 统一token过期时间为2小时
   - [ ] 修复刷新token的race condition
   - [ ] 添加更详细的日志
   
4. 用户确认
5. 执行修复
6. 验证修改

用户：测试通过，部署上线
```

## 本章彩蛋

### Cursor Agent的隐藏技巧

**1. 快捷键**：
- `Cmd/Ctrl + I`：快速打开Agent模式
- `@`：在对话中引用文件、代码块
- `@codebase`：引用整个代码库

**2. 特殊指令**：
```
/show plan - 显示当前计划
/revise - 修改计划
/skip - 跳过当前步骤
/rollback - 回滚修改
```

**3. MCP配置示例**：
```json
{
  "mcpServers": {
    "database": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "DATABASE_URL": "postgresql://..."
      }
    }
  }
}
```

### Agent模式的边界

即使是最先进的Agent模式，也有明确边界：

1. **不能替代架构决策**
   - AI可以执行，但重大架构决策需要人工判断

2. **不能保证业务正确性**
   - AI不懂你的业务规则，生成的代码需要业务验证

3. **不能承担安全责任**
   - 安全相关代码必须人工审查

4. **不能处理未知问题**
   - 遇到意外情况，AI会停下来询问

记住：**你是指挥官，AI是执行者**。指挥得当，效率翻倍；盲目信任，事故难免。
