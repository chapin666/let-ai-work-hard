# 第5章：让AI一步步思考复杂问题

> **提示工程进阶——CoT与结构化输出**

---

## 故事：小陈的"算法困境"

### 周一：一个看似简单的需求

小陈是公司的新晋开发工程师，入职刚满一年。虽然他工作很努力，但面对复杂问题时常常感到力不从心。

周一早晨，技术负责人老李给了他一个任务：

"小陈，我们系统需要一个智能排班功能。要根据员工的可用时间、技能标签、工作负载，自动生成一周的排班表。你研究一下怎么实现。"

小陈听了有点懵。排班？这不是简单的轮班表，而是要考虑各种约束条件的最优化问题。

他打开Cursor，试着问AI：

```
实现一个智能排班系统，根据员工可用时间、技能标签、工作负载自动生成排班表
```

AI很快给出了回答，是一段简单的循环代码：

```javascript
// AI生成的简单实现
function generateSchedule(employees, shifts) {
  const schedule = {};
  
  for (let day = 0; day < 7; day++) {
    schedule[day] = {};
    
    for (const shift of shifts) {
      // 找第一个可用的员工
      const availableEmployee = employees.find(e => 
        e.availableDays.includes(day) && 
        e.skills.includes(shift.requiredSkill)
      );
      
      if (availableEmployee) {
        schedule[day][shift.id] = availableEmployee.id;
      }
    }
  }
  
  return schedule;
}
```

小陈看着这段代码，总觉得哪里不对。这个实现太简单了——

- 没有考虑公平性（有人可能一周都是夜班）
- 没有处理冲突（两个人被排到同一个班次）
- 没有考虑工作负载均衡
- 没有处理无解的情况

他试着追问AI："这个实现没有考虑公平性和负载均衡，能改进吗？"

AI又生成了一段代码，加了一些简单的计数逻辑，但核心问题依然存在。

"看来AI也不是万能的..."小陈有些沮丧。

周三下午，小陈在茶水间碰到了小李——那个用AI用得飞起的后端工程师。

"李哥，我想请教个问题。"小陈把自己的困境说了一遍。

"什么意思？"

"你有没有试过**Chain of Thought**？让AI一步步分析问题，把思路理清楚，最后再写代码。"

小陈摇摇头。

"排班是个典型的约束满足问题，你应该先让AI分析：

2. 可以用什么算法解决？
3. 每种算法的优缺点是什么？
4. 针对你的场景应该选择什么方案？

等思路清晰了，再让它写代码。"

小陈眼睛一亮："原来还可以这样！"

### 周四：CoT实战

周四，小陈决定用Chain of Thought方法重新尝试。

他没有直接要求写代码，而是让AI先分析问题：

```
我需要实现一个智能排班系统。请先分析这个问题，然后给出解决方案。

业务背景：
- 客服团队有20名员工
- 每天3个班次：早班(8-16)、中班(12-20)、晚班(16-24)
- 每个班次需要2-3人
- 员工有不同的技能标签（如：英语、技术、投诉处理）
- 员工每周最多工作5天
- 员工可以设置不可用时间

请按以下步骤分析：
1. 识别所有约束条件（硬约束和软约束）
2. 分析可能的算法方案（贪心、回溯、遗传算法等）
3. 评估每种方案的优缺点和适用场景
4. 推荐最适合的方案并说明理由
5. 给出实现思路（不需要完整代码）
```

这次AI的回答完全不同了：

**步骤1：约束条件分析**

硬约束（必须满足）：
- 每个班次人数在2-3人之间
- 员工每周工作不超过5天
- 员工不能被排在自己不可用的时间段
- 每个班次至少有一名具备必要技能的员工

软约束（尽量满足）：
- 工作负载均衡（每人工作时间相近）
- 班次类型均衡（不要总排某人夜班）
- 技能覆盖最大化

**步骤2：算法方案对比**

| 算法 | 优点 | 缺点 | 适用场景 |
|:---|:---|:---|:---|
| 贪心算法 | 快，简单 | 可能不是最优解 | 约束简单，速度要求高 |
| 回溯算法 | 能找到可行解 | 慢，复杂度高 | 小规模问题（<10人）|
| 遗传算法 | 能处理复杂约束 | 实现复杂，参数调优难 | 大规模，多目标优化 |
| 线性规划 | 数学最优 | 需要专业求解器 | 纯数学问题 |
| 启发式搜索 | 平衡效果与性能 | 需要设计启发函数 | 中等规模实际问题 |

**步骤3：推荐方案**

对于20人规模的客服排班，建议采用**启发式搜索 + 局部优化**的混合方案：

1. 先用贪心算法生成初始解
2. 用局部搜索（如模拟退火）优化软约束
3. 如果找不到可行解，反馈冲突信息给用户

**步骤4：实现思路**

```
数据结构设计：
- Employee: {id, skills[], maxDays, unavailable[]}
- Shift: {id, day, type, requiredSkills[], minStaff, maxStaff}
- Schedule: Map<shiftId, employeeId[]>

算法流程：
1. 初始化空排班表
2. 按优先级排序班次（难排的先排）
3. 对每个班次：
   - 筛选可用员工
   - 按负载均衡选择员工
4. 检查约束，标记违规
5. 用交换/替换操作优化
6. 返回排班表和违规报告
```

小陈看着这个分析，思路一下子清晰了。他继续追问：

```
基于以上分析，请实现核心排班算法。

要求：
1. TypeScript实现
2. 包含完整的类型定义
3. 有清晰的注释说明算法逻辑
4. 包含简单的测试用例
5. 如果无法找到可行解，返回冲突信息而不是null
```

这次生成的代码质量明显高了很多——结构清晰，考虑了各种约束，还有错误处理。

## 理论知识：Chain of Thought与结构化输出

### 什么是Chain of Thought（CoT）？

Chain of Thought（思维链）是一种让AI**逐步推理**的技术。

**核心思想**：不是直接要答案，而是让AI展示思考过程。

**效果对比**：

| 方法 | 提示词 | 结果 |
|:---|:---|:---|
| 直接提问 | "小明有5个苹果，给了小红2个，又买了3个，现在有几个？" | "6个"（可能正确，也可能错误） |
| CoT | "...请一步步计算" | "初始5个→给小红后5-2=3个→买了3个后3+3=6个，答案6个"（正确率更高） |

**为什么CoT有效？**

1. **分解复杂问题**：把大问题拆成小步骤
2. **减少错误传播**：每一步都可以验证
3. **提高可解释性**：能看到AI的推理过程
4. **允许修正**：发现某步错了，可以针对性调整

### CoT的三种形式

#### 1. Zero-Shot CoT（零样本思维链）

最简单的形式，在提示词中加入"一步步思考"的指令。

```
问题：优化这个SQL查询

SELECT * FROM orders o 
JOIN users u ON o.user_id = u.id 
WHERE u.status = 'active'

请一步步分析这个查询的性能问题，然后给出优化方案。
```

#### 2. Few-Shot CoT（少样本思维链）

提供一个示例，展示期望的思考方式。

```
以下是问题分析和解决的示例：

问题：网站加载很慢，如何优化？

分析过程：
1. 先确定慢的原因：打开Chrome DevTools的Network面板
2. 发现主要是图片加载慢：图片平均大小2MB
3. 解决方案：压缩图片、使用WebP格式、启用CDN
4. 验证：优化后图片平均大小200KB，加载时间从3s降到0.5s

现在请用同样的方式分析我的问题：
[你的问题]
```

#### 3. Tree of Thoughts（思维树）

对于更复杂的问题，可以让AI探索多个可能的思路。

```
我需要设计一个缓存策略。请从以下几个角度分析，给出每种方案的优缺点：

思路1：本地内存缓存（如Node-cache）
思路2：分布式缓存（如Redis）
思路3：CDN缓存
思路4：浏览器缓存

对于每种思路，请分析：
- 适用场景
- 实现复杂度
- 性能表现
- 一致性保证

最后根据我的场景（高并发读、低频写、数据一致性要求不高）给出推荐方案。
```

### 结构化输出

结构化输出是指让AI按照**预定义的格式**返回结果。

#### 为什么需要结构化输出？

1. **便于解析**：程序可以直接处理，不需要额外解析
2. **确保完整**：不会遗漏字段
3. **类型安全**：配合TypeScript等类型系统
4. **便于验证**：可以定义schema进行校验

#### 常用格式

**JSON格式**：
```
请将分析结果输出为以下JSON格式：
{
  "summary": "简要总结",
  "issues": [
    {
      "severity": "high|medium|low",
      "description": "问题描述",
      "suggestion": "修复建议"
    }
  ],
  "score": 0-100的评分
}
```

**Markdown表格**：
```
请将对比结果输出为Markdown表格：
| 方案 | 优点 | 缺点 | 推荐指数 |
```

**XML格式**：
```
请用XML格式输出：
<analysis>
  <issue severity="high">...</issue>
</analysis>
```

### CoT + 结构化输出的组合使用

这两种技术结合使用效果最佳：

```
我需要分析一段代码的质量。请按以下步骤进行：

步骤1：代码结构分析
- 识别主要函数和模块
- 分析依赖关系

步骤2：潜在问题识别
- 检查常见反模式
- 评估性能和安全性

步骤3：改进建议
- 给出具体的优化方案

请以以下JSON格式输出结果：
{
  "structure": {
    "modules": ["模块名"],
    "dependencies": ["依赖项"]
  },
  "issues": [
    {
      "type": "performance|security|maintainability",
      "line": 行号,
      "description": "问题描述",
      "suggestion": "修复建议"
    }
  ],
  "improvements": ["改进建议列表"]
}

代码：
[粘贴代码]
```

## 实践案例：复杂任务的CoT模板

### 模板1：系统设计任务

```
我需要设计一个[系统名称]。请按以下步骤分析：

步骤1：需求澄清
- 核心功能是什么？
- 用户是谁？使用场景是什么？
- 预期的规模和性能要求？

步骤2：约束分析
- 技术约束（现有技术栈、团队技能）
- 业务约束（合规、安全、成本）
- 时间约束

步骤3：方案设计
- 提供2-3个可行的架构方案
- 每个方案包含：架构图描述、技术选型、优缺点

步骤4：方案推荐
- 根据约束条件选择最佳方案
- 说明选择理由

步骤5：实施计划
- 分阶段实施建议
- 风险点和应对措施

请以结构化格式输出，便于后续讨论。
```

### 模板2：代码审查任务

```
请审查以下代码，按步骤分析：

步骤1：理解代码意图
- 这段代码要解决什么问题？
- 核心逻辑是什么？

步骤2：结构分析
- 函数/类的划分是否合理？
- 依赖关系是否清晰？

步骤3：问题识别
检查以下方面：
- 代码规范（命名、格式）
- 潜在Bug（空指针、边界情况）
- 性能问题（算法复杂度、不必要的计算）
- 安全问题（注入、XSS等）
- 可维护性（注释、测试、文档）

步骤4：改进建议
对每个问题给出：
- 优先级（P0/P1/P2）
- 具体修复方案
- 修复后的代码示例

输出格式：
```json
{
  "summary": "总体评价",
  "score": 1-10,
  "issues": [
    {
      "priority": "P0|P1|P2",
      "category": "bug|performance|security|style",
      "line": 行号,
      "description": "问题描述",
      "suggestion": "修复建议",
      "fixed_code": "修复后的代码"
    }
  ]
}
```

代码：
[粘贴代码]
```

## 本章彩蛋

### CoT的"元提示"技巧

你可以让AI自己生成CoT步骤：

```
我有一个问题：[问题描述]

请帮我设计一个分析这个问题的步骤清单。要求：
1. 步骤要具体可执行
2. 每个步骤有明确的产出
3. 步骤之间有逻辑递进关系
4. 最后一步是具体的输出格式

然后按照这个清单帮我分析问题。
```

### 自我验证技巧

让AI自己检查自己的答案：

```
请回答以下问题，然后自我验证：

问题：...

回答：...

验证：请检查上述回答是否有错误，列出可能的错误点和修正建议。
```

### 多视角分析

让AI从多个角度分析问题：

```
请从以下三个角色的角度分析这个方案：

1. 开发工程师：关注实现难度、代码质量
2. 产品经理：关注用户需求、功能完整性
3. 运维工程师：关注部署、监控、故障处理

每个角色给出：
- 最担心的1-2个问题
- 最认可的1-2个优点
- 1条改进建议
```
