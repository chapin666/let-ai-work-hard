# 第27章：把"做个电商后台"变成清晰PRD

> **AI辅助产品经理工作流——需求分析篇**

---

## 故事：阿强的需求困境

### 周一：那个令人窒息的会议

"阿强，这个电商后台的需求什么时候能给我？"

项目经理老李站在阿强的工位旁，手里拿着一杯咖啡，眼神里带着那种"我知道你很忙但我也没办法"的歉意。

阿强抬起头，看了看电脑屏幕上那行写了三天的标题——《电商后台管理系统需求文档》。三天了，除了标题，正文一个字都没动。

"我...正在梳理，"阿强说，"这周能给你初稿。"

老李点点头走了，但阿强知道他心里在想什么——"又在画饼了"。

不能怪老李。过去三个月，阿强负责的三个项目都延期了。不是因为开发慢，而是需求阶段就出问题了。

**第一次**：需求评审会上，开发刚看了5分钟就炸了——"这个'用户管理'功能，到底包含哪些操作？增删改查？批量操作？权限控制？你这一句话我怎么估时？"

**第二次**：设计稿都快画完了，阿强才发现漏了一个关键场景——"第三方登录失败后的降级方案是什么？"

**第三次**：项目进行到一半，业务方突然说"不对，我们还需要一个拼团功能"——而这个需求在最初的需求调研里完全没有体现。

"我是不是不适合做产品？"阿强盯着屏幕上那个孤零零的标题，陷入了深深的自我怀疑。

---

### 周二：一个偶然的机会

周二下午，阿强去参加了一个产品经理的内部分享会。分享者是公司的高级产品经理林姐，据说她带的项目从来不需要返工。

"我的秘诀？"林姐笑了笑，"用AI当我的需求分析搭档。"

会议室里一片哗然。

"AI？那不是写代码用的吗？"

"AI能写PRD？那要我们产品经理干嘛？"

林姐摆摆手："你们理解错了。AI不是替代我们做需求分析，而是帮我们做那些**耗时但不需要创造力**的工作——比如结构化梳理、边界情况检查、关联功能分析。"

她展示了一个案例。

**原始需求**："做一个支持多种促销方式的电商后台。"

这是她给AI的输入。然后AI输出的内容让她自己都吃惊：

**功能模块拆解**：
- 促销规则管理（增删改查）
- 促销类型定义（满减、折扣、买一送一、限时购等）
- 促销适用范围（全店、指定商品、指定类目）
- 促销时间控制（开始/结束时间、周期性促销）
- 促销叠加规则（是否可叠加、叠加优先级）
- 促销效果预览（模拟计算促销后价格）
- 促销数据统计（参与人数、销售额、ROI）

**边界情况检查**：
- 多个促销同时生效时的优先级冲突
- 促销商品库存不足的处理
- 促销期间商品下架的补偿机制
- 用户取消订单后的促销资格恢复
- 促销与会员折扣的叠加逻辑

**关联功能影响**：
- 订单模块：需要支持促销金额拆分
- 库存模块：需要预留促销库存
- 支付模块：需要显示促销后价格
- 物流模块：促销赠品需要单独处理
- 财务模块：需要促销成本核算

"看到了吗？"林姐说，"这些分析不是AI拍脑袋想出来的，而是基于电商行业的通用模式。我只需要在这个基础上，结合我们公司的具体业务场景做调整。"

"以前我做这个分析，可能要翻遍竞品、查资料、脑暴会议，至少需要两三天。现在用AI，30分钟出初稿，1小时精修，效率提升10倍。"

阿强的眼睛亮了。

他想起自己那个写了三天还是只有标题的PRD。如果AI能帮他做结构化的需求分析...

---

### 周三：第一次AI辅助需求分析

周三早上，阿强决定实战一下。

他打开ChatGPT，没有像以前写PRD那样直接"开干"，而是先做了一个**需求输入的准备工作**。

**第一步：明确背景和目标**

```
我需要为一个B2C电商平台设计后台管理系统的需求文档。

背景信息：
- 平台主营3C数码产品
- 目前年GMV约5000万
- 运营团队10人，客服团队15人
- 当前使用第三方SaaS系统，功能受限
- 计划6个月内完成自建系统并切换

目标：
- 支持商品、订单、库存、用户、营销五大模块
- 支持多角色权限（运营、客服、财务、管理员）
- 与现有ERP、物流系统对接
- 系统需要支撑未来3年业务增长

请帮我进行全面的需求分析，输出完整的PRD框架。
```

**第二步：让AI扮演不同角色进行审查**

阿强记得林姐分享的一个技巧——让AI扮演不同角色来审查需求，可以发现不同视角的问题。

```
现在请你分别扮演以下角色，对上述需求提出问题和建议：

1. 后端开发工程师：关注技术可行性、数据模型、接口设计
2. 前端开发工程师：关注交互复杂度、页面性能、组件复用
3. 测试工程师：关注可测试性、边界情况、验收标准
4. 运营人员：关注日常使用流程、操作效率、数据报表
5. 财务人员：关注资金流转、对账、结算

每个角色提出3-5个关键问题。
```

AI给出的回答让阿强倒吸一口凉气。

**后端开发视角**：
- 商品SKU和SPU的数据模型如何设计？属性系统是否支持动态扩展？
- 订单状态机有多少种状态？状态流转是否有幂等性保证？
- 库存扣减是下单扣还是支付扣？高并发场景下的超卖如何处理？
- 与ERP、物流系统的对接是实时同步还是批量同步？失败如何重试？

**前端开发视角**：
- 商品列表页预期多少数据量？是否需要虚拟滚动或分页？
- 订单详情页需要展示多少信息？是否需要分Tab展示？
- 后台管理系统是否响应式？支持移动端访问吗？
- 表单页面是否有自动保存草稿功能？防止误操作丢失数据。

**测试工程师视角**：
- 每个功能点的验收标准是什么？如何定义"完成"？
- 是否需要进行压力测试？预期的并发量是多少？
- 多角色权限测试的矩阵是怎样的？每个角色能做什么、不能做什么？
- 数据迁移方案是什么？如何验证迁移的正确性？

**运营人员视角**：
- 商品上架流程需要几步？能否批量操作？
- 订单处理能否批量发货、批量打印快递单？
- 日常需要看哪些数据报表？能否自定义？
- 遇到用户投诉，客服能否快速查询订单、处理退款？

**财务人员视角**：
- 订单的资金流如何与财务系统对账？
- 退款是原路退回还是退到余额？财务如何核算？
- 平台佣金、商家结算的周期和规则是什么？
- 是否需要导出符合会计准则的财务报表？

阿强看着这些问题，额头开始冒汗。

这些问题，如果他直接写PRD，大概率会漏掉一大半。而一旦漏掉，就意味着开发阶段返工、测试阶段扯皮、上线后运营抱怨。

"原来需求分析不是'我想清楚功能'就够了，还要考虑这么多维度的影响..."

**第三步：结构化输出PRD框架**

现在阿强对需求的复杂度有了更清醒的认识。他让AI输出一个完整的PRD框架：

```
基于以上分析，请生成一份完整的PRD框架，包含以下内容：

1. 文档信息（版本、作者、更新记录）
2. 项目背景与目标
3. 用户角色与权限矩阵
4. 功能模块详细说明（每个模块包含：功能描述、业务流程、页面原型描述、验收标准）
   - 商品管理
   - 订单管理
   - 库存管理
   - 用户管理
   - 营销管理
5. 非功能性需求（性能、安全、兼容性）
6. 数据埋点需求
7. 项目风险与应对
8. 附录（术语表、参考文档）

要求：
- 每个功能点都要具体、可验收
- 使用表格、列表等结构化方式呈现
- 标注需要进一步确认的问题
```

AI生成的PRD框架让阿强惊讶不已。虽然具体的业务逻辑还需要他填充，但整个文档的结构、需要考虑的维度、甚至一些行业通用的最佳实践都已经有了雏形。

---

### 周四：填充细节与迭代优化

有了框架，接下来就是填充细节。阿强发现，AI在这方面同样能帮上大忙。

**场景1：写用户故事**

阿强需要为"商品上架"功能写用户故事，但总是写不好。他尝试让AI辅助：

```
请为"商品上架"功能写3个用户故事，格式：
作为[角色]，我希望[功能]，以便[价值]

要求：
1. 覆盖主要场景（单个上架、批量上架、快速复制上架）
2. 体现不同角色的需求差异
3. 价值描述要具体，不要泛泛而谈
```

AI输出：

**用户故事1：单个商品上架**
- 作为运营专员，我希望通过一个表单完整录入商品信息（基本信息、规格、价格、库存、图文详情），以便将新采购的商品快速上架销售。

**用户故事2：批量商品上架**
- 作为运营主管，我希望通过Excel导入的方式一次性上架多个商品，以便在季度上新或促销备货时提高效率，减少重复操作。

**用户故事3：复制商品上架**
- 作为运营专员，我希望能够复制已有商品的信息并稍作修改后快速上架，以便针对相似商品（如同一型号不同颜色）减少重复录入工作。

阿强在这些基础上做了微调，加入了一些公司的具体业务规则，很快就完成了这个模块的用户故事。

**场景2：梳理业务流程**

对于订单处理流程，阿强一直理不清楚状态流转。他让AI帮忙：

```
请帮我梳理电商订单的完整业务流程，包括：

1. 订单状态定义（列出所有可能的状态）
2. 状态流转图（用Mermaid语法）
3. 每个状态变更的触发条件
4. 异常分支处理（如取消、退款、超时关闭）

要求考虑以下业务场景：
- 正常流程：下单→支付→发货→收货→完成
- 退款流程：申请→审核→退款→完成
- 异常流程：超时未支付、发货超时、拒收
```

AI不仅给出了清晰的流程图，还列出了他没想到的一些状态，比如"待补货"（库存不足时的临时状态）、"风控审核中"（大额订单的安全检查）。

**场景3：检查遗漏**

写完一个模块后，阿强会让AI帮他检查是否遗漏了什么：

```
我已经完成了"订单管理"模块的PRD，主要包含：
- 订单列表（查询、筛选、导出）
- 订单详情（基本信息、商品信息、物流信息、操作日志）
- 订单操作（发货、取消、退款、备注）

请帮我检查：
1. 还缺少哪些常见的订单管理功能？
2. 这些功能是否有遗漏的边界情况？
3. 与其他模块的关联是否考虑周全？
```

AI的反馈：

**可能遗漏的功能**：
- 订单合并（同一用户的多个订单合并发货）
- 订单拆分（一个订单分多个包裹发货）
- 改价功能（运营后台直接修改订单金额）
- 订单标签/标记（方便运营分类管理）
- 批量操作（批量发货、批量打印面单）

**边界情况**：
- 部分退款（只退订单中的某几个商品）
- 换货流程（不退款只换商品）
- 预售订单（定金+尾款模式）
- 虚拟商品订单（无需物流，自动发货）

**关联模块考虑**：
- 与库存模块：发货时如何扣减库存？占用库存和可用库存的区别？
- 与营销模块：使用了优惠券的订单退款，优惠券是否返还？
- 与会员模块：订单完成后积分如何计算？退货后积分是否扣回？

阿强看着这些反馈，默默地在自己的PRD里增加了一个"待补充"列表。

### 周五：评审会上的逆袭

周五下午，阿强拿着写好的PRD参加需求评审会。

他有点紧张。以前的需求评审，总是被开发怼得哑口无言。这次会不一样吗？

会议开始。

后端负责人老王接过PRD，扫了一眼目录，眼神变了："嗯，结构挺清晰的。"

他翻到"订单管理"模块："这个状态机定义得挺详细的，不过我想确认一下——'已支付'到'待发货'的流转，是由支付回调触发的还是定时任务轮询的？"

阿强心里一喜——这个问题他在写PRD的时候就考虑过。"由支付回调触发，同时有一个兜底的重试机制。详见第3.2.4节'支付状态同步'。"

老王点点头，继续看。

前端负责人小李翻到"商品管理"："商品表单挺复杂的，有没有字段说明？"

"有，"阿强翻到附录，"这里是完整的字段清单，包含字段名、类型、必填/选填、校验规则、示例值。"

测试工程师小张在边上插话："这个字段清单太有用了，我可以直接拿来设计测试用例。"

会议进行了两个小时，出乎意料地顺利。虽然也有一些需要调整的地方，但都是"优化建议"级别的，而不是"你这个需求没法做"级别的。

会议结束时，老王拍了拍阿强的肩膀："这个PRD写得专业，以后都按这个标准来。"

阿强走出会议室，长舒一口气。

---

### 周五晚上：复盘与总结

晚上，阿强坐在电脑前，回顾这一周的经历。

他打开自己的笔记，写下了这段经历的总结：

**以前写PRD的问题**：
1. 想到哪写到哪，结构混乱
2. 只考虑功能本身，不考虑关联影响
3. 遗漏边界情况，开发阶段才发现
4. 验收标准模糊，测试阶段扯皮
5. 一个PRD写一周，效率低下

**用AI辅助后的改变**：
1. AI帮我建立结构，我填充业务逻辑
2. AI帮我多维度检查，我确认合理性
3. AI帮我列举边界情况，我判断优先级
4. AI帮我生成验收标准，我调整细节
5. 两天完成高质量PRD，效率提升3倍

"AI不是替代我做需求分析，"阿强写道，"它帮我做了那些繁琐但重要的'体力活'，让我能把精力放在真正需要思考和判断的地方。"

## 理论知识：AI辅助需求分析方法论

### 传统需求分析 vs AI辅助需求分析

| 维度 | 传统方式 | AI辅助方式 |
|:---|:---|:---|
| **需求拆解** | 人工 brainstorm，耗时且易遗漏 | AI基于行业模式快速生成框架 |
| **多维度检查** | 依赖经验，新人容易漏 | AI扮演多角色自动审查 |
| **边界情况** | 靠测试阶段发现 | AI前置分析，提前暴露 |
| **文档结构** | 每次从零开始 | AI生成模板，保持一致性 |
| **效率** | 3-5天/份PRD | 1-2天/份高质量PRD |

### AI辅助需求分析的4层模型

#### 第1层：框架生成——从0到1的结构搭建

**核心作用**：基于行业通用模式，快速生成需求文档的完整框架。

**适用场景**：
- 新项目的PRD初稿
- 不熟悉领域的快速入门
- 确保文档结构的完整性

**使用方法**：
```
我需要设计一个[系统/功能]的需求文档。

背景：
[业务背景、用户群体、核心痛点]

目标：
[期望达成的业务目标、成功指标]

请基于行业最佳实践，生成完整的PRD框架，包含：
1. 功能模块清单
2. 每个模块的核心功能点
3. 需要考虑的边界情况
4. 与其他系统的关联点
```

#### 第2层：多角色审查——发现盲点

**核心作用**：让AI扮演不同利益相关方，提前发现需求盲点。

**适用场景**：
- PRD初稿完成后的自检
- 需求评审前的预审
- 复杂功能的全面分析

**使用方法**：
```
请扮演以下角色，对我的需求文档提出问题和建议：

1. 后端开发：关注数据模型、接口设计、技术可行性
2. 前端开发：关注交互复杂度、页面性能、组件复用
3. 测试工程师：关注可测试性、边界情况、验收标准
4. 运营/业务方：关注日常使用、操作效率、数据需求
5. 安全/合规：关注数据安全、权限控制、合规风险

每个角色提出3-5个关键问题，并说明为什么这个问题重要。
```

#### 第3层：细节填充——从1到100的完善

**核心作用**：在框架基础上，填充具体的业务逻辑和细节。

**适用场景**：
- 用户故事撰写
- 业务流程梳理
- 字段定义和校验规则
- 验收标准编写

**使用方法**：
```
请帮我完善[功能模块]的详细需求：

功能概述：
[一句话描述功能]

用户场景：
[描述典型使用场景]

请输出：
1. 详细的业务流程（含正常流程和异常分支）
2. 页面原型描述（包含哪些元素、如何布局）
3. 字段清单（字段名、类型、必填、校验规则）
4. 验收标准（可测试的完成标准）
```

#### 第4层：质量检查——确保完整性

**核心作用**：在PRD完成前，进行全面的质量检查。

**适用场景**：
- PRD定稿前的最终检查
- 需求变更后的影响分析
- 确保跨模块的一致性

**使用方法**：
```
请对我的PRD进行全面的质量检查：

[粘贴PRD全文或关键部分]

检查维度：
1. 完整性：是否遗漏了常见的功能点？
2. 一致性：术语、定义是否统一？
3. 可验收性：每个需求是否有明确的验收标准？
4. 可行性：是否存在明显的技术或业务不可行点？
5. 关联性：是否考虑了与其他模块的交互？

输出格式：
- [问题级别] 问题描述 → 建议修改方案
```

### 案例：设计一个"会员积分系统"

**Step 1：框架生成**

```
我需要设计一个电商平台的会员积分系统，请生成完整的需求框架。

背景：
- 现有用户200万，月活50万
- 计划通过积分系统提升用户粘性和复购率
- 需要支持积分获取、消耗、过期全生命周期管理

目标：
- 积分获取率（获得积分的用户/总用户）>30%
- 积分消耗率（消耗积分/发放积分）>60%
- 会员复购率提升10%

请输出：
1. 功能模块清单
2. 每个模块的核心功能
3. 需要考虑的边界情况
4. 与现有系统的集成点
```

**Step 2：多角色审查**

```
基于以上框架，请扮演以下角色提出问题：

1. 后端开发：积分并发发放如何处理？积分和金额的换算精度？
2. 前端开发：积分展示的场景有哪些？实时性要求？
3. 运营：如何配置积分规则？能否A/B测试？
4. 财务：积分是否构成负债？如何核算成本？
5. 风控：如何防止积分套利？
```

**Step 3：细节填充**

```
请帮我详细设计"积分获取规则配置"功能：

功能描述：
运营后台可以灵活配置用户获得积分的规则

请输出：
1. 规则类型（购物返积分、签到、邀请好友等）
2. 每种规则的配置字段
3. 规则的生效/失效机制
4. 用户端的展示逻辑
```

**Step 4：质量检查**

```
请对完整的积分系统PRD进行质量检查，重点关注：
1. 积分计算的准确性（精度、并发）
2. 积分过期的处理（提醒、清零逻辑）
3. 退款的积分回退（部分退款如何处理）
4. 跨平台一致性（APP、小程序、H5）
```

## 本章彩蛋

### 彩蛋1：3个让AI输出更精准的秘诀

**秘诀1：提供充足的背景信息**

❌ 不好的输入：
"帮我写一个商品管理功能的需求。"

✅ 好的输入：
"我需要为一个主营服装的B2C电商平台设计商品管理功能。平台SKU数量约5000，每月新增300个SKU。运营团队5人，需要支持批量操作。请基于这些信息进行需求分析。"

**秘诀2：明确输出格式和详细程度**

❌ 不好的输入：
"帮我分析需求。"

✅ 好的输入：
"请按以下格式输出需求分析：
1. 功能清单（使用表格，包含优先级）
2. 边界情况（使用列表，标注严重程度）
3. 关联影响（使用思维导图格式）
要求每个功能点都具体到可验收的程度。"

**秘诀3：迭代优化，而不是一次到位**

❌ 不好的做法：
期望AI一次输出完美的PRD。

✅ 好的做法：
第一轮：生成框架 → 人工审查调整
第二轮：填充细节 → 人工补充业务规则
第三轮：检查完善 → 人工确认最终版本

### 彩蛋2：需求分析的隐藏Prompt

**Prompt 1：反向思考**
```
如果我完全不做这个功能，用户会怎样解决他们的问题？
现有解决方案的缺点是什么？
这个功能可能会带来什么负面效果？
```

**Prompt 2：极端场景**
```
请列出这个功能可能遇到的所有极端场景：
- 数据量极端（极大或极小）
- 用户行为极端（频繁操作、恶意操作）
- 系统状态极端（网络中断、服务降级）
```

**Prompt 3：竞品对标**
```
淘宝/京东/拼多多是怎么做这个功能的？
它们各自有什么优缺点？
我们应该参考哪一个？为什么？
```
